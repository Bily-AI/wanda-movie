# 万达快速出票第四阶段设计：支付前置数据与订单查询

## 背景

前三个阶段已经跑通了主购票链路：本地应用壳、万达账号登录、城市/影院/影片/日期/场次查询、动态座位图、确认选座、创建订单和取消订单。第四阶段继续以旧 Electron 打包产物为准，复用旧万达接口和当前 Vue 3 + Electron 技术选型，不 mock 数据。

本阶段选择较稳的切片：补齐创建订单后的支付前置数据和历史订单查询。它为后续真实支付、支付宝跳转、自动支付打基础，但本阶段不触发真实扣款。

## 目标

1. 创建电影票订单后，能查询并展示真实的支付活动、可用支付卡、可用兑换券和订单状态。
2. 购票页右侧“全局订单信息 / 支付活动 / 支付卡 / 兑换券 / 已选座位”使用真实订单上下文更新。
3. 历史订单页使用旧万达订单接口加载真实订单，并支持现有筛选、刷新、导出入口。
4. 接口失败、账号未选、订单未创建等场景不生成假数据，只显示空状态、错误提示和日志。

## 明确不做

1. 不发起真实支付，不调用支付宝自动支付，不完成扣款。
2. 不实现自动支付设备指纹登录、滑块处理或验证码绕过。
3. 不重做卡券、会员、活动页面的完整业务，只为购票支付前置数据和订单查询补必要能力。
4. 不使用 mock 数据填充页面；接口没有返回时保持空状态。
5. 不替换当前技术栈，不引入新的请求体系或大范围 UI 重构。

## 旧接口依据

旧打包产物中可用的第四阶段接口集中在 `seatApi` 相关模块，本阶段只接与支付前置和订单查询相关的接口：

1. 支付活动：`/mkt/activity/secret/list.api`、`/mkt/activity/secret/ncoupons.api`、`/mkt/activity/secret/conponuse.api`
2. 支付卡：`/card/pay/list.api`
3. 兑换券：`/coupon/member/grouplist.api`
4. 订单状态：`/order/order_status.api`
5. 历史订单：`/order/query_order_list.api`
6. 支付信息查询：`/order/query_pay_info_upgrade.api`

这些接口继续通过当前 `wandaCore` 的 host、签名、请求头和 cookie 机制发送。旧产物里已经存在的 Baidu OCR、支付宝网关和自动支付能力不在本阶段启用。

## 页面设计

### 购票页

购票页保留现有截图布局，不重画页面结构。

订单创建成功后，页面保存当前订单上下文，包括账号、手机号、城市、影院、影片、日期、场次、座位、订单号和金额。右侧区域根据订单上下文刷新：

1. “全局订单信息”展示当前订单摘要；没有订单时继续显示“暂无订单”。
2. “支付活动”加载活动列表，默认保留“无活动”；接口返回可用活动时允许选择。
3. “支付卡”加载当前账号可用支付卡，显示卡名、卡号、余额、状态和可选数量。
4. “兑换券”加载券列表，并根据接口结果区分可用、不可用、已选和需要数量。
5. “已选座位”继续展示当前已选座位，订单创建后补充订单关联信息。

底部“提交支付”在本阶段改为支付前检查入口：检查当前是否已有订单、是否已加载支付前置数据，并查询订单状态。它不调用预支付、合并支付或支付宝网关。

### 历史订单页

历史订单页接真实订单列表：

1. 使用当前选中的万达账号查询订单。
2. 支持关键词、订单状态、日期范围和刷新。
3. 汇总今日订单、待处理、已完成和总金额。
4. 列表字段沿用现有表头：手机号、订单号、影片、影院、场次、金额、状态、创建时间、操作。
5. “操作”先提供查看支付信息/刷新状态类动作，不提供再次支付或退款。
6. 导出基于当前已加载数据生成，不额外 mock。

## 数据流

1. 用户选择万达账号并完成城市、影院、影片、日期、场次和座位选择。
2. 用户确认选座后，当前已有流程创建订单。
3. 创建订单成功后，订单上下文进入购票状态，并触发支付前置数据刷新。
4. 支付活动、支付卡、兑换券分别调用旧接口加载，转换为页面需要的轻量结构。
5. 用户点击“提交支付”时，只做订单状态查询和前置数据完整性检查。
6. 历史订单页进入或刷新时，根据当前账号和筛选条件查询真实订单列表。

## 状态边界

1. 未选择账号：购票页和历史订单页提示先选择账号，不请求接口。
2. 未创建订单：支付活动、支付卡、兑换券保持空状态或默认“无活动”。
3. 接口失败：保留已有数据或清空对应区域，并记录错误日志。
4. 账号切换：清空当前订单和支付前置数据，避免跨账号串单。
5. 取消订单成功：清空当前订单关联的支付前置数据，并刷新订单状态。
6. 重复点击：支付前置查询和订单状态查询要有 loading 防重复。

## 代码设计

沿用当前代码组织，优先局部扩展，不新增大而全的抽象。

1. `src/shared/wandaCore.ts`
   - 补齐本阶段需要的接口路径或复用已存在路径。
   - 保持 host、签名、form/json 请求方式与旧接口一致。

2. `src/renderer/services/seatApi.ts`
   - 增加支付活动、支付卡、兑换券、订单状态、历史订单和支付信息查询方法。
   - 返回值做最小标准化，保留必要原始字段，方便后续真实支付阶段继续使用。

3. `src/renderer/stores/ticket.ts`
   - 增加当前订单、支付活动、支付卡、兑换券、订单状态等状态。
   - 提供刷新支付前置数据、清空订单支付上下文、查询订单状态的方法。

4. `src/renderer/views/TicketView.vue`
   - 接入新状态并渲染右侧区域。
   - 保留当前座位图和下单流程，不改动动态座位布局逻辑。
   - “提交支付”只执行前置检查和订单状态查询。

5. `src/renderer/stores/orders.ts` 与 `src/renderer/views/OrderHistoryView.vue`
   - 接入真实历史订单查询。
   - 复用现有筛选和汇总 UI，补齐数据转换和导出。

6. `scripts/check-phase4.*`
   - 新增第四阶段检查脚本，覆盖接口参数、数据转换、订单状态和页面状态边界。

## 错误处理和日志

1. 用户可理解的错误通过页面提示展示，例如“请先选择账号”“订单状态查询失败”。
2. 技术细节写入日志页或开发日志，不把接口原始异常直接铺到 UI。
3. 对真实订单、卡券和活动数据不做兜底模拟，避免误导出票判断。
4. 支付相关按钮在请求中禁用，防止重复查询或重复状态变更。

## 测试和验收

1. 单元检查：验证支付活动、支付卡、兑换券、订单状态、历史订单的数据转换。
2. 状态检查：验证账号切换、订单取消、无订单、接口失败时清理正确。
3. 页面检查：验证购票页右侧区域和历史订单页在有/无数据时显示合理。
4. 回归检查：继续运行已有 `check:renderer`、`check:phase2`、`check:phase3`、`typecheck` 和 `build`。
5. 新增 `check:phase4`，作为本阶段验收入口。

## 后续阶段预留
