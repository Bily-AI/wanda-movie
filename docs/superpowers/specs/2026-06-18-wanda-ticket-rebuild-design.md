# 万达快速出票 1.0.0 重构设计

## 目标

重新开发以前打包后的万达购票工具，作为可维护的 `1.0.0` 桌面应用。新应用以你提供的旧版截图为产品原型，保留旧技术选型，并继续使用真实的万达、百度 OCR、支付宝相关接口。业务数据不做 mock。

## 已确认决策

- 新项目版本从 `1.0.0` 开始。
- 当前阶段不使用旧的远程软件鉴权服务器。
- 软件登录、注册、鉴权暂不做。
- 应用启动后直接进入主界面。
- 万达账号登录仍然保留，因为万达业务接口依赖 CK、token、userIdentifier 等账号信息。
- 你提供的截图作为首版 UI 和流程原型。
- 座位图必须来自万达实时座位接口，不能写成本地固定模板。
- 后续文档、必要代码注释、commit message 均使用中文。
- Electron 打包产物是重构事实源。旧包里能复用的接口、签名、设备参数、IPC、样式、资源、页面结构、本地数据格式，优先从旧包提取和沿用。

## 1.0.0 不做的内容

- 不接新的远程授权服务器。
- 不做本地软件鉴权登录页。
- 不 mock 城市、影院、影片、座位、订单、卡券、储值卡、会员、活动等业务数据。
- 不重新设计一套和截图差异很大的产品布局。
- 不为了少量差异新增无收益的公共抽象、公共组件分支或配置映射。
- 不在旧包已有可用实现时，另造一套新的接口协议、签名规则、座位解析、账号数据结构或页面交互。

## 旧打包产物复用原则

旧 Electron 打包产物优先级高于主观重写。重构的目标是把旧包里已经跑通过的逻辑恢复成可维护源码，而不是凭空设计一套新系统。

优先复用范围：

- `resources/app/out/main/index.js` 中的窗口、IPC、HTTP 转发、百度 OCR、支付宝转换、剪贴板、截图、本地文件读写逻辑。
- `resources/app/out/preload/index.js` 中暴露给渲染进程的 API 形状。
- `resources/app/out/renderer/assets` 中的页面分块、路由、Pinia store、万达接口、签名、请求头、座位渲染、卡券/会员/活动逻辑。
- `config/accounts.json`、`config/city.json`、`config/proxy.json` 等本地数据格式和迁移线索。
- 旧截图中的布局、文案、控件顺序、按钮位置、状态提示。
- 旧包里的图片、图标、样式片段、颜色和交互细节，能直接提取时优先使用。

需要重写的情况：

- 旧代码被混淆或压缩到不可维护，只能提取行为和参数后重新组织源码。
- 旧实现依赖已经废弃的远程软件鉴权服务器。
- 旧实现存在明显坏配置或错误，例如坏 JSON，需要兼容修复。
- 为了适配新工程结构，需要把散落逻辑整理到服务模块，但外部行为保持旧版一致。

每个模块开工前先查旧包对应实现，再决定是直接复用资源、迁移逻辑，还是按旧行为重写。

## 技术选型

沿用旧版技术路线：

- Electron：桌面壳、窗口控制、IPC、本地文件、截图、剪贴板、支付窗口。
- Vue 3：渲染进程 UI。
- Vue Router：页面路由。
- Pinia：账号、购票、设置、日志、主题等状态。
- Element Plus 和 `@element-plus/icons-vue`：基础 UI 控件。
- Axios 或主进程 HTTP 转发：网络请求。
- CryptoJS 和 JSEncrypt：万达签名、加密和旧逻辑兼容。
- `pinyin-pro`：城市、影院、影片搜索。
- `qrcode`：支付或链接二维码。
- `docx`：保留旧版导出相关能力。

## 总体架构

### 主进程

主进程负责：

- 创建无边框主窗口，并保留旧版窗口行为。
- 提供 `wanda-http-get`、`wanda-http-post`、debug 请求等 IPC。
- 读写本地 JSON：账号、城市缓存、代理设置、卡券分类、设置、日志。
- 处理百度 OCR 请求。
- 处理截图、剪贴板图片、剪贴板文本。
- 将支付宝 SDK 或订单支付参数转换为 H5 支付链接。
- 管理独立的支付宝会话和可选的设备同步。
- 预留未来软件鉴权 IPC 边界，但 `1.0.0` 默认不启用。

### 渲染进程

渲染进程负责：

- 按旧截图复刻可操作界面。
- 左侧万达账号面板常驻。
- 顶部页面：购票、历史订单、储值卡、兑换券、会员、活动、日志、设置。
- 组织用户选择流程并调用万达业务服务。
- 根据实时接口数据渲染座位图。
- 展示错误、状态和操作日志。

### 服务模块

建议拆分边界：

- `services/wanda/core`：主机、盐值、时间戳、设备参数、ShumeiBoxId、请求头、万达转义。
- `services/wanda/user`：万达短信验证码、登录、登录态检查、CK 和 userIdentifier。
- `services/wanda/cinema`：城市缓存、影院详情、影院场次、影片/日期/场次整理。
- `services/wanda/seat`：实时座位、座位解析、选座限制。
- `services/wanda/order`：创建订单、取消、退款、状态、历史订单、支付信息、预支付、合并支付。
- `services/wanda/payment`：支付活动、支付可用卡券、储值卡、支付宝转换。
- `services/wanda/card`：储值卡列表、购买、充值、赠送。
- `services/wanda/coupon`：绑定卡券、卡券列表、分类、赠送、生成链接。
- `services/wanda/member`：Rtime/W+ 会员、签到、等级权益。
- `services/wanda/activity`：影院礼包活动、礼包详情、礼包下单。
- `services/ocr`：百度 OCR 和识别结果整理。
- `services/local`：本地账号、设置、日志、分类、代理配置。

页面组件只负责 UI 和用户流程。签名、接口路径、请求头生成放在服务层。

## UI 原型映射

### 启动与主框架

应用直接进入主界面。旧版登录/注册卡片暂不出现。

主框架按截图复刻：

- 顶部显示应用名和版本 `v1.0.0`。
- 顶部导航：购票、历史订单、储值卡、兑换券、会员、活动、日志、设置。
- 右上角保留应用状态、版本或退出入口。
- 优先实现浅色主题；深色主题如果沿用旧逻辑成本可控，则保留设置项。

### 左侧账号面板

保留：

- 分组选择。
- 刷新和搜索。
- 账号表格：手机号、备注、状态。
- 当前使用账号高亮。
- 导入账号。
- 移动到分组。
- 取消选择。
- 万达账号登录框：手机号、验证码、登录。

这里是万达业务账号登录，不是软件本身鉴权。

### 购票页面

按截图 3-5 复刻：

- 左侧：万达账号列表和账号登录。
- 中上：搜索、城市、影院、影片、日期、场次、刷新座位。
- 中下：动态座位图。
- 底部：刷新取票码、图片识别、文本识别。
- 右侧：全局订单信息、支付活动、支付卡、兑换券、已选座位。
- 右下：取消选择、确认选座、提交支付。

选择流程：

1. 从真实城市缓存选择或搜索城市。
2. 选择或搜索影院。
3. 调万达接口获取影院场次。
4. 选择影片。
5. 选择日期。
6. 选择场次。
7. 调万达实时座位接口。
8. 渲染接口返回的座位布局。
9. 选座，并按需加载活动、支付卡、卡券。
10. 创建订单并进入支付。

座位图要求：

- 不硬编码排、列和影厅布局。
- 保留已售、可选、已选、不同区域/票种颜色。
- 大厅支持横向和纵向滚动。
- 显示已选座位数和最大座位限制。
- 刷新座位不丢失当前万达账号。

### 其他页面

历史订单：

- 支持手机号、订单号、影片搜索。
- 支持状态筛选、日期范围、刷新、导出。
- 保留今日订单、待处理、已完成、总金额统计卡片。
- 表格列按旧截图复刻。

储值卡：

- 按当前万达账号展示储值卡。
- 展示持有人、卡名称、卡号、余额、状态、操作。
- 保留购买、充值、详情、赠送入口。

兑换券：

- 保留绑定卡券、批量赠送、分类筛选、券名筛选、关键词搜索、统计、刷新。
- 保留生成链接和赠送流程。

会员：

- 保留 Rtime 和 W+ 页签。
- 展示会员等级、签到、成长进度、权益、领取/使用操作。

活动：

- 保留城市/影院选择。
- 展示可购买礼包和我的礼包订单。
- 保留代理 API、代理选择相关入口。

日志：

- 保留本地操作日志。
- 支持日志类型、账号、详情、日期筛选和清空日志。

设置：

- 主题设置。
- 窗口位置恢复。
- 购票设置。
- 自动支付设置。
- 业务请求头参数和设备参数。
- 代理 API 设置。
- 缓存清理。
- 关于信息显示 `1.0.0` 和 `Electron + Vue 3`。

## OCR 与识别

保留旧版百度 OCR 流程：

- 截取指定区域或读取剪贴板图片。
- 主进程向百度 OCR 发送图片 base64。
- 渲染进程接收 OCR words。
- 保留图片识别和文本识别按钮。

OCR 后处理可以保留旧版结构化解析能力，但必须隔离在 `services/ocr` 后面，方便后续替换或关闭。

## 支付

保留：

- 支付活动选择。
- 储值卡使用。
- 兑换券使用。
- 订单预支付和合并支付。
- 支付宝参数转换 H5 链接。
- 独立支付宝会话和清理会话。
- 设置页里的自动支付入口。

支付失败时保留订单信息并允许重试。除非用户明确点击取消，不自动取消订单。

## 本地数据

本地数据尽量使用可读 JSON：

- `accounts.json`：万达账号、分组、CK、userIdentifier、状态、备注。
- `city.json`：城市和影院缓存，可先从旧包导入，后续再更新。
- `proxy.json`：代理 API 和开关。新应用必须兼容空配置或坏 JSON。
- `categories.json`：卡券分类。
- `settings.json`：UI、窗口、支付、业务请求参数。
- `logs.json` 或本地日志存储：操作日志。

迁移逻辑应读取旧包配置，并规范化到新应用数据目录。

## 错误处理

- 万达登录过期：标记账号异常，提示重新登录。
- 万达签名或设备参数被拒：显示请求类型和账号，并写入诊断日志。
- 刷新座位失败：保留选择控件，清空过期座位图。
- 创建订单失败：显示万达返回消息，保留已选座位直到用户清空或刷新。
- 支付转换失败：保留订单信息并允许重试。
- OCR 失败：显示可读的百度错误，允许手动输入文本。
- 本地配置损坏：自动用默认值修复，并尽量备份旧文件。

## 测试与验证

因为必须使用真实接口，验证分阶段进行：

1. 静态验证：应用能构建和启动。
2. 本地 UI 验证：所有路由按旧截图结构渲染。
3. 账号验证：万达短信登录能保存 CK 和 userIdentifier。
4. 选择链路验证：城市、影院、影片、日期、场次均来自真实数据。
5. 座位验证：实时座位图会随影院和场次变化，并正确统计数量。
6. 订单验证：使用可控测试账号创建、取消、查询订单。
7. 支付验证：预支付和支付宝转换能进入预期流程。
8. 卡券/储值卡/会员/活动验证：每个页面能用当前账号加载真实数据。
9. OCR 验证：剪贴板或图片识别能返回文字。
10. 旧包对照验证：关键接口参数、请求头、座位解析和页面行为要和旧打包产物逐项对照。

## 实施阶段

### 阶段 1：脚手架与桌面壳

创建干净的 Electron + Vue 3 项目，保留旧版窗口风格、主进程 IPC、路由、Pinia、Element Plus，并设置版本 `1.0.0`。开工前先建立旧包索引，记录可复用文件、接口、资源和本地配置。

### 阶段 2：本地数据与万达核心

实现本地配置、账号 store、万达签名/请求头、设备参数、HTTP 桥接。

### 阶段 3：万达账号登录与购票主链路

实现万达短信登录、城市/影院/影片/日期/场次联动、实时座位图、选座、创建/取消订单。

### 阶段 4：支付与 OCR

实现支付活动、支付卡、兑换券、支付宝转换、截图/剪贴板、百度 OCR 和 OCR 解析接入。

### 阶段 5：剩余页面

实现历史订单、储值卡、兑换券、会员、活动、日志、设置。

### 阶段 6：加固

补充配置迁移、错误修复、日志、重试和逐路由验证。

## 未来软件鉴权

应用预留鉴权边界，但默认关闭：

- `auth` store 占位。
- 路由守卫占位。
- 登录路由占位。
- 主进程鉴权 IPC 占位。

以后加软件鉴权时，只负责控制是否进入应用，不改万达服务模块和页面业务流程。

## 风险

- 万达部分接口可能已经和旧包时期不一致。
- 旧 CK 可能过期或绑定设备。
- OCR 和 AI 解析凭据可能失效，后续应重新配置或轮换。
- 支付宝转换依赖旧行为，需要真实支付参数验证。
- 旧包代码被混淆，后续不能盲目搬运，需要逐接口验证。

## 1.0.0 验收标准

- 应用直接进入主界面，并显示版本 `1.0.0`。
- UI 尽量贴近你提供的旧版截图。
- 真实万达账号可以登录并被选中。
- 城市、影院、影片、日期、场次、座位均来自旧万达接口。
- 座位布局会随影院和场次动态变化。
- 下单、取消、查询、支付准备、储值卡、卡券、会员、活动均使用真实接口。
- 百度 OCR 通过主进程工作。
- 业务页面不依赖 mock 数据。
- 软件鉴权暂不出现，但后续可以通过预留边界加回来。
