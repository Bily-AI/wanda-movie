# 万达快速出票 1.0.0 第三阶段设计

## 背景

第一阶段已经完成 Electron + Vue 3 工程骨架和旧截图页面骨架。第二阶段已经完成本地状态、本地 JSON 存储、受控输入、设置保存、万达 HTTP IPC 边界和敏感信息扫描。

第三阶段进入主业务链路：万达账号短信登录、城市/影院/影片/日期/场次联动、实时座位图、选座、电影票下单和取消订单。本阶段继续以旧 Electron 打包产物为事实源，业务数据必须来自旧万达接口，不写 mock 城市、影院、影片、座位或订单数据。

软件自身的登录、注册、远程鉴权和本地鉴权继续不做。应用启动后仍直接进入主界面；这里的“登录”只指万达业务账号登录。

## 方案选择

采用方案 1：分两步跑通第三阶段。

第一步先接真实短信登录、城市/影院/影片/日期/场次联动、实时座位和本地选座，确保用户可以用真实账号看到真实座位图。第二步在同阶段末尾接电影票下单和取消订单，并在 UI 上保留明确确认，避免误下单。

不采用一次性全链路并行实现，因为真实接口调试、签名、设备参数、座位变化和订单状态都可能同时失败，定位成本高。不采用只做接口封装和页面占位，因为这不能满足“主功能跑通”和“不 mock 数据”的要求。

## 范围

包含：

- 万达账号验证码发送、验证码登录、登录状态检查。
- 登录成功后保存 `ck`、`userIdentifier`、手机号、会员标识和账号状态。
- 使用本地真实城市/影院数据，缺失时显示清晰错误，不补 mock 数据。
- 影院场次接口接入，生成影片、日期、场次选择项。
- 影院详情接口接入，用于补充影院地址等展示信息。
- 实时座位接口接入，按返回坐标动态渲染座位图。
- 本地选座、取消选择、已选座位列表、总价计算。
- 电影票创建订单和取消订单。
- 接口失败、登录失效、座位过期、下单失败写入本地日志。
- 关键链路契约检查、类型检查、构建检查和敏感信息扫描。

不包含：

- 支付活动、支付卡、兑换券抵扣、预支付、合并支付和支付宝转换。
- 百度 OCR、图片识别、文本识别真实链路。
- 历史订单、储值卡、兑换券、会员、活动页面的完整真实数据补齐。
- 软件自身鉴权。
- 自动下单、自动支付、无确认创建订单。

## 旧接口对齐

第三阶段只记录接口形状，不记录旧包里的签名敏感值。

万达账号接口：

- `POST /user/login_verify_code.api`
  - body：`phone`
  - 成功依据：返回 `msg` 为成功文案且存在 `requestID`。
- `POST /user/login.api`
  - body：`cinemaId`、`userPlat`、`phone`、`vcode`、`requestId`
  - 成功依据：`code === 0` 且存在 `data.userToken`
  - 保存：`data.userToken` 作为 `ck`，同时保存 `data.userIdentifier`。
- `POST /user/islogin.api`
  - body：`json=true`
  - 成功依据：`data.success === true`
  - 更新：手机号、`userIdentifier`、`isPayMember` 和账号状态。

影院与场次接口：

- `GET /showtime/by_cinema.api`
  - query：`cinemaId`、`showType=0`、`json=true`
  - 数据入口：`data.showtimeFilmInf`
  - 展开顺序：影片、日期、场次。
- `GET /cinema/by_cinemaid.api`
  - query：`cinemaid`
  - 注意旧字段拼写可能是 `cinemaDetial`，新工程需要兼容。

座位与订单接口：

- `GET /order/real_time_seat.api`
  - query：`dId`
  - 数据入口：`data.realtimeSeats.area`
- `POST /order/create_order.api`
  - body：`retailerCode=MX`、`mobile`、`seatId`、`totalPrice`、`dId`
  - 多个 `seatId` 使用旧包相同的分隔与编码方式。
  - 成功依据：`code === 0 && data.bizCode === 0`
  - 返回：`data.orderId`
- `POST /order/cancel.api`
  - body：`orderId`
  - 成功依据：`code === 0 && data.bizCode === 0`

注意：旧包里的 `/order/create.api` 是卡券或储值卡相关下单，不作为电影票下单接口。

## 架构

沿用第二阶段工程结构，不新增框架。

- `src/shared` 放跨进程类型、接口常量、请求校验和座位数据结构。
- `src/main` 保持本地文件读写和 HTTP 转发，不把页面流程写进主进程。
- `src/preload` 继续暴露受控 IPC 方法。
- `src/renderer/stores` 维护账号、购票、座位、订单和日志状态。
- `src/renderer/views` 只组织页面交互和展示。
- `src/renderer/components` 承载可复用但具体的座位图、已选座位列表等局部组件。

建议新增或补齐的模块：

- `src/shared/seatTypes.ts`：实时座位返回、座位区域、原始座位、归一化座位类型。
- `src/renderer/services/wandaAuthApi.ts`：验证码、登录、登录状态检查。
- `src/renderer/services/cinemaApi.ts`：影院场次、影院详情。
- `src/renderer/services/seatApi.ts`：实时座位、电影票下单、取消订单。
- `src/renderer/components/SeatMap.vue`：坐标座位图。
- `src/renderer/components/SelectedSeatList.vue`：已选座位列表。

如果当前文件已有同类写法，优先按现有写法补局部逻辑，不为了少量差异新建过多抽象。

## 请求边界

第二阶段的 `wanda-http-get` 和 `wanda-http-post` 继续作为唯一网络出口。第三阶段需要兼容旧包 POST 表单体，所以请求校验允许 body 为普通对象或字符串。

请求安全规则：

- 只允许旧万达 host。
- 只允许已登记的旧接口路径。
- 缺少手机号、验证码、`requestId`、`ck`、`userIdentifier`、`cinemaId`、`dId`、`seatId` 等必要参数时，不发送请求。
- 签名、设备参数、请求头由旧包行为迁移得到，但敏感值不进入文档、测试样例或提交历史。
- 真实账号、CK、代理、OCR key 和签名敏感值只能存在于本地配置或运行时状态中。

## 数据流

万达账号登录：

1. 用户输入手机号并点击获取验证码。
2. 调用验证码接口，保存本次 `requestID` 到登录表单状态。
3. 用户输入验证码并点击登录。
4. 调用登录接口，成功后写入账号 store 和本地 JSON。
5. 选中当前账号，后续购票请求使用该账号的 `ck` 和 `userIdentifier`。
6. 登录状态检查失败时，把账号标记为异常并提示重新登录。

购票联动：

1. 从真实城市/影院数据中选择或搜索城市。
2. 根据城市筛选影院。
3. 选择影院后调用影院场次接口。
4. 从场次返回中生成影片列表。
5. 选择影片后生成可用日期。
6. 选择日期后生成场次列表。
7. 选择场次后保存 `dId`，清空旧座位和旧选座。
8. 点击刷新座位或自动加载实时座位。

选座与订单：

1. 实时座位返回后，按 `area[].seat[]` 归一化为本地座位节点。
2. 座位图只用接口返回的 `coordx`、`coordy`、`row`、`column`、`seatId`、`areaId`、`status` 和价格信息。
3. `status === 1` 视为可选，其余视为不可选或已售。
4. 点击可选座位加入本地已选，再次点击取消。
5. 确认选座只做本地确认和右侧已选座位展示。
6. 提交支付按钮在本阶段创建电影票订单，但点击前必须二次确认。
7. 创建订单成功后展示订单号和订单状态；支付能力留到第四阶段。
8. 用户明确点击取消订单时调用取消订单接口。

## 座位渲染

座位图必须动态渲染，不能按截图写固定排列表。

旧包座位布局规则：

- 单个座位宽高约为 `26 x 22`。
- 行标按 `coordy` 去重排序。
- 横坐标由 `coordx` 计算，纵坐标由 `coordy` 计算。
- 大厅支持横向和纵向滚动。
- 屏幕线显示在座位图上方。
- 图例保留普通区、优选区、VIP 区、W+会员区、情侣区、特惠区。

区域颜色沿用旧截图和旧包样式。若旧接口返回未知 `areaCode`，默认按普通区展示，但保留原始区域字段，避免丢失服务端信息。

## 错误处理

- 验证码发送失败：展示万达返回文案，不清空手机号。
- 登录失败：展示万达返回文案，不写入 CK。
- 登录失效：账号状态改为异常，购票请求提示重新登录。
- 城市或影院数据缺失：显示“缺少真实城市/影院数据”，不填 mock。
- 场次加载失败：保留已选城市/影院，清空影片、日期、场次。
- 座位加载失败：清空过期座位图，保留当前场次，允许重试。
- 下单失败：保留已选座位，优先显示 `bizMsg`，其次显示接口 `msg`。
- 取消订单失败：保留订单状态，记录日志。
- 请求参数不完整：本地拦截并提示缺少字段。

## 日志

日志只记录必要诊断信息：

- 操作类型。
- 账号手机号脱敏值。
- 影院、影片、场次、座位数量。
- 接口路径。
- 成功或失败文案。
- 时间。

日志不记录完整 CK、完整 userIdentifier、签名值、验证码、代理账号密码和任何密钥。

## 测试与验证

自动验证：

- 类型检查。
- 构建检查。
- 渲染契约检查。
- 第三阶段接口契约检查。
- 敏感信息扫描。

手动验证：

- 输入手机号可以获取验证码。
- 输入验证码可以登录真实万达账号并保存账号。
- 选择城市和影院后，影片、日期、场次来自真实接口。
- 不同影院或场次的座位图会变化。
- 可选座位可以选择和取消。
- 已选座位数量、座位号和总价正确。
- 创建订单需要明确确认。
- 创建订单成功后可以看到订单号。
- 取消订单只在用户明确操作时发生。

真实下单验证必须使用用户可控账号和可控场次，不做自动化真实下单。

## 验收标准

- 应用仍显示 `v1.0.0`，仍按旧截图结构进入主界面。
- 万达账号短信登录可用，登录成功后账号列表状态变为正常。
- 购票页城市、影院、影片、日期、场次联动来自真实接口或真实本地数据。
- 实时座位图来自 `/order/real_time_seat.api`，座位不是固定模板。
- 本地选座、取消选择和已选座位面板可用。
- 电影票创建订单使用 `/order/create_order.api`。
- 取消订单使用 `/order/cancel.api`。
- 不出现 mock 城市、影院、影片、座位、订单数据。
- 不提交真实账号、CK、验证码、密钥、签名敏感值或代理凭证。
- 类型检查、构建检查和阶段契约检查通过。
