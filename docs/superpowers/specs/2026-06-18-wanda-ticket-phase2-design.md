# 万达快速出票 1.0.0 第二阶段设计

## 背景

第一阶段已经完成 Electron + Vue 3 工程骨架、页面路由、主界面布局和旧截图原型复刻。当前页面仍是静态骨架，很多输入框没有绑定本地状态，部分选择框按截图初始态被禁用，所以用户现在看到“输入框不能输入”是阶段一未接交互导致的现象，不是最终行为。

第二阶段目标是让主功能页面具备真实可交互基础，并为后续接入旧万达接口、百度 OCR 和真实座位链路打好边界。软件自身鉴权继续跳过，万达账号登录能力保留到第三阶段接真实接口。

## 范围

第二阶段只做本地状态、本地数据、万达核心请求边界和输入可用性，不做完整业务链路跑通。

包含：

- 购票、账号侧栏、历史订单、储值卡、兑换券、会员、活动、日志、设置页的输入控件接入本地状态。
- 账号、分组、当前账号、设置、代理、请求参数、日志等本地 JSON 读写边界。
- 旧包可复用配置结构的迁移入口，优先兼容旧包 `config` 数据格式。
- 万达接口 host、路径、请求头、设备参数、签名入口和主进程 HTTP 桥接。
- 防止真实账号、CK、代理、OCR key、AI key、签名敏感值被写入提交内容。
- 检查脚本覆盖关键交互状态、IPC 边界和敏感信息泄漏。

不包含：

- 软件自身登录、注册、远程鉴权或本地鉴权。
- 万达短信验证码发送和登录真实跑通。
- 城市、影院、影片、日期、场次真实联动。
- 实时座位图、选座、下单、取消、支付。
- 百度 OCR 真请求。
- 历史订单、卡券、储值卡、会员、活动页真实数据加载。

这些内容保留到第三阶段及后续阶段实现。

## 方案选择

### 方案 A：先做输入可用和本地状态

优点是最快解决当前“不能输入”的问题，页面马上能进入可操作状态。缺点是如果不同时建立数据边界，后续接接口时还要返工。

### 方案 B：先做接口核心，再回填页面交互

优点是优先对齐旧万达接口。缺点是用户侧还会继续看到页面不可输入，阶段反馈不明显。

### 方案 C：本地状态和接口边界并行推进

推荐采用。先让输入框、选择器和设置项可编辑，同时在主进程建立本地数据和万达请求桥接。这样既解决当前可用性问题，又不把页面做成 mock 数据驱动。

## 架构

第二阶段沿用第一阶段工程结构，不引入新的前端框架。

- `src/main` 负责本地 JSON 文件读写、旧配置迁移、HTTP 桥接和敏感能力。
- `src/preload` 暴露受控 IPC API，保持渲染进程不能直接访问 Node 能力。
- `src/shared` 放置跨进程类型、常量和纯函数。
- `src/renderer/stores` 保存页面状态、账号状态、设置状态和日志状态。
- `src/renderer/views` 只做 UI 和流程绑定，避免把签名、请求头和本地文件逻辑写进页面。

页面输入框以 `v-model` 绑定 Pinia store 或局部表单状态。真正依赖上游选择的控件仍保留禁用逻辑，例如未选择影院前不能选影片，未选择影片前不能选日期，未选择日期前不能选场次。

## 本地数据

本地数据使用可读 JSON，主进程统一读写，渲染进程通过 IPC 调用。

计划维护的数据：

- `accounts.json`：分组、万达账号、备注、状态、CK、userIdentifier、当前选择。
- `settings.json`：主题、窗口记忆、购票设置、支付显示方式、取票码面板模板、自动支付开关。
- `request-params.json`：设备指纹、设备型号、用户标识、ShumeiBoxId 等业务请求头参数。
- `proxy.json`：代理 API、代理开关、活动页代理选择。
- `logs.json`：本地操作日志。
- `city.json`：旧包迁移或后续接口缓存的城市和影院数据。

真实账号、CK、代理、密钥类数据只允许存在用户本地数据目录或被 `.gitignore` 忽略的本地文件中，不写进仓库文档和测试夹具。

## 万达请求核心

第二阶段只建立边界，不承诺真实业务接口全部成功。

需要落地：

- 旧版 host 常量。
- 旧版 API 路径常量。
- 统一请求方法、query/body 排序和时间戳处理入口。
- 请求头生成入口。
- 设备参数读取和刷新入口。
- 签名函数入口。
- `wanda-http-get` 和 `wanda-http-post` IPC。

签名敏感值、OCR key、AI key 不写入文档。实现时优先从旧包可迁移配置、用户本地配置或环境变量读取；如果缺失，返回明确错误，提示需要补充本地配置，而不是静默使用假数据。

## UI 交互

第二阶段要修复当前输入体验：

- 购票页手机号、验证码、搜索、城市、影院等输入可编辑。
- 设置页代理 API、自动支付账号、支付密码、请求参数等输入可编辑。
- 日志页关键词、日期、类型筛选可编辑。
- 活动页城市、影院、礼包 ID、代理 API 可编辑。
- 历史订单页搜索、状态、日期筛选可编辑。
- 卡券、储值卡、会员页保留当前截图结构，接入当前账号和筛选状态。

没有真实接口数据前，不填充业务 mock 列表。空状态继续显示“暂无数据”或旧截图里的默认提示。

## 错误处理

- 本地 JSON 不存在时创建默认结构。
- 本地 JSON 损坏时备份原文件，再用默认结构恢复。
- 旧配置迁移失败时记录日志，不中断应用启动。
- 万达请求参数不完整时返回清晰错误，不发送半成品请求。
- 敏感配置缺失时提示“需要本地配置”，不写入源码默认值。

## 验证

第二阶段完成前需要验证：

- 页面关键输入框可以输入并保留状态。
- 依赖上游选择的控件只在缺少前置条件时禁用。
- 本地 JSON 读写 IPC 可用。
- 万达 HTTP 桥接 IPC 存在并有参数校验。
- 构建和类型检查通过。
- 仓库中不包含真实账号、CK、代理、密钥、签名敏感值。

计划运行：

- 渲染契约检查。
- 本地数据契约检查。
- 类型检查。
- 构建检查。
- 敏感信息扫描。

## 验收标准

- 用户在主界面能正常输入手机号、验证码、搜索词、设置项和筛选项。
- 页面不会再出现“普通输入框完全无法输入”的体验问题。
- 空数据状态仍然不是 mock 数据。
- 主进程具备本地数据读写和万达请求桥接边界。
- 后续第三阶段可以直接在这些边界上接万达短信登录、城市影院影片联动和实时座位。
